# Coding Conventions

**Analysis Date:** 2026-03-25

## Naming Patterns

**Files:**
- Header files: `*.h` with PascalCase (e.g., `DmaDriver.h`, `AxisDriver.h`, `AxiVersion.h`)
- Source files: `*.c` for kernel drivers, `*.cpp` for applications
- Test files: `*Test.cpp` or `test.cpp` in app directories

**Functions:**
- PascalCase for exported functions (e.g., `dmaRead`, `dmaWrite`, `dmaInitMaskBytes`)
- `AxisG1_*` and `AxisG2_*` prefix for hardware-specific implementations
- `Gpu_*` prefix for GPU-related functions
- `Dma_*` prefix for common driver functions

**Variables:**
- camelCase for local variables (e.g., `dmaIndex`, `rxData`, `buff`)
- `g` prefix for global variables (e.g., `gDmaDevCount`, `gCl`)
- `dev->` prefix for device structure members
- UpperSnakeCase for constants (e.g., `DMA_VERSION`, `DMA_ERR_FIFO`)

**Types:**
- `struct` names in PascalCase (e.g., `struct DmaDevice`, `struct DmaBuffer`)
- `typedef` not commonly used; direct `struct` references preferred

## Code Style

**Formatting:**
- Indentation: 3 spaces (configured in `.editorconfig`)
- No tabs (enforced by CI pipeline)
- Maximum line length: 250 characters (configured in `CPPLINT.cfg`)

**Linting:**
- **cpplint** - Python-based C++ linter
  - Config: `CPPLINT.cfg`
  - Run via: `find . -name '*.h' -o -name '*.cpp' -o -name '*.c' | xargs cpplint`
  - Disabled checks:
    - `-legal/copyright` - License header checks
    - `-build/header_guard` - Header guard style
    - `-readability/casting` - C-style casts in kernel code
    - `-runtime/int` - `unsigned long` usage in kernel APIs
    - `-runtime/threadsafe_fn` - Known unsafe thread usage in apps
    - `-build/include_subdir` - Headers in same directory
    - `-whitespace/indent` - 3-space indentation (conflicts with `.editorconfig`)

- **clang-tidy** - Static analysis tool
  - Config: `.clang-tidy`
  - Uses `compile_commands.json` generated by `bear`
  - Enabled checks: `bugprone-*`, `performance-*`
  - Disabled: Various checks specific to kernel development (int-to-ptr casting, etc.)

## Import Organization

**Order:**
1. System headers (`<stdio.h>`, `<stdlib.h>`, `<linux/...>`)
2. Project headers (`<DmaDriver.h>`, `<AxisDriver.h>`, etc.)

**Path Aliases:**
- None explicitly configured; uses relative paths within project

## Error Handling

**Patterns:**
- Functions return negative values or `-1` on error (e.g., `return -1;`, `return -ENOMEM;`)
- `dev_err()`, `dev_warn()`, `dev_info()` for kernel logging
- User-space applications use `printf()` for error messages
- Error codes: `DMA_ERR_FIFO`, `DMA_ERR_LEN`, `DMA_ERR_MAX`, `DMA_ERR_BUS`

## Logging

**Framework:** Kernel: `printk` variants; Userspace: `printf`

**Patterns:**
- Kernel: `dev_<level>(dev->device, ...)` where level is `err`, `warn`, `info`
- Include context: buffer sizes, addresses, flags
- Debug logging controlled by `dev->debug` flag

## Comments

**When to Comment:**
- All source files start with multi-line header comments (company, description, license)
- Functions have Doxygen-style comments describing parameters, return values
- Complex logic blocks have inline comments explaining purpose

**JSDoc/TSDoc:**
- Not applicable; uses Doxygen-style `@param`, `@return` tags in C comments

## Function Design

**Size:** Functions vary from inline helpers to ~200-line drivers; no strict limit

**Parameters:**
- Pointers first for device structures, then configuration parameters
- Common pattern: `dev`, `buff`, `count`, `flags` ordering

**Return Values:**
- Kernel: `int`/`ssize_t` returning 0 on success, negative on error
- Userspace: `int` returning 0 on success, non-zero on error

## Module Design

**Exports:**
- Kernel: `struct file_operations` for driver interface
- Userspace: Static inline functions in header files

**Barrel Files:**
- `include/DataDriver.h` - Main header that includes all other headers
